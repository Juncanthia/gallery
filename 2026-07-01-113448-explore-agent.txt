     edit
     extend 目录 Base UI → Radix 迁移计划

     Context

     extend(src/components/extend/)是站内唯一还在用 @base-ui/react
     的组件目录,其余组件库早已统一到 radix-ui。目标是让 extend 也去掉
     @base-ui/react 依赖,统一技术栈。

     前序已完成两轮扫描 +
     逐文件实读(不是凭经验推断),确认了几个会实际影响执行方式的事实:

     1. 实际需要动的文件只有 15 个(extend/components/ui/ 下 43 个文件里,只有这
     15 个 import 了 @base-ui/react),不是最初粗估的 21 个。
     2. extend/components/ui/ 
     里的组件全部是内部实现细节,没有一个作为独立组件被写进 
     content/components/extend/*.mdx 文档。该目录下只有 13 篇 mdx,全部对应
     docx-editor / excel-editor / csv-viewer / pdf-viewer 等"业务级 block",没有
     button.mdx / dialog.mdx 这类原语文档。
     → 结论:15 个原语文件的对外 API(props 命名、导出的 CreateHandle
     等)不是公开契约,只要保证这 13 个 block
     的外观和交互行为不变,内部实现可以放开改。
     3. PopoverCreateHandle / DialogCreateHandle / TooltipCreateHandle
     三个导出全仓库 0 处调用(grep CreateHandle 只在定义处命中),docx-editor.tsx
     真正用到的是 PreviewCardPrimitive.createHandle(第 736 行附近,按段落样式 id
     命令式打开预览卡片)。→ 这三个未用到的 handle 可以直接去掉,不用做兼容层。
     4. src/primitives/radix/ 下已有 accordion / dialog / dropdown-menu /
     hover-card / popover / sheet / tabs / collapsible / toggle / toggle-group /
     tooltip 的 Radix 实现,但它们是主系统"炫技"版本,统一套了 motion/react
     做动画、getStrictContext+useControlledState 做受控状态。extend 自己完全是
     Tailwind + Base UI 原生的 data-starting-style/data-ending-style(CSS
     @starting-style)/data-instant 驱动动画,没有引入 motion。因此
     src/primitives/radix/* 只能当"这个组件在 Radix
     下由哪些子组件构成"的参考,不能直接复制——直接抄会把 motion 意外引入
     extend,风格也对不上。真正贴近 extend 现有风格的参考是
