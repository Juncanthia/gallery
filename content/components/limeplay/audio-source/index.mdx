AudioSourceProvider 音频源提供者。用于向 LimePlay 媒体播放器 Store 注入音频资源数据，支持单源、多源数组和带专辑/艺术家元数据的复杂音频资产，是音频播放器生态的数据入口组件。

<h2 id="when-to-use">何时使用</h2>

当需要向 LimePlay 播放器上下文注入音频资源时使用。AudioSourceProvider 内部通过 PlaybackSourceController 处理资源的解析、加载和回放控制，支持 `src` 直链和 `playbackUrls` 懒加载两种模式。

常见场景：

- 音频播放器中加载一个或多个音频文件。
- 音乐播放应用中注入带专辑、艺术家、封面等元数据的曲目。
- 配合 `usePlaylist` 和 `usePlaybackStore` 实现播放列表管理。
- 在 `MediaProvider` 上下文中作为数据入口，供其他控件读取资源状态。

不推荐：

- 在非 `MediaProvider` 上下文之外独立使用 AudioSourceProvider，它依赖 `usePlaybackStore` 和 `usePlaylistStore`。
- 视频播放场景 —— 应使用 `VideoPlayer` 或 `MediaPlayer` 组件的 `source` 属性。
- 纯前端的播放列表 UI 展示 —— AudioSourceProvider 是数据层，不提供 UI 渲染。

<h2 id="examples">代码演示</h2>

### 基础用法

通过 `source` 传入 `AudioPlayerAsset` 数组，AudioSourceProvider 自动将资源注入播放 Store。子组件可通过 `usePlaylistStore` 读取播放队列。

<ComponentShowcase id="limeplay/audio-source" path="limeplay/audio-source/basic" />

```tsx
import { AudioSourceProvider } from "@/components/ui/audio-source"

const DEMO_SOURCE = [
  {
    id: "demo-track-1",
    src: "https://example.com/audio-1.mp3",
    title: "Sample Track 1",
    artistName: "Artist Name",
    albumName: "Album Title",
    genre: "Pop",
    releaseYear: 2024,
  },
  {
    id: "demo-track-2",
    src: "https://example.com/audio-2.mp3",
    title: "Sample Track 2",
    artistName: "Artist Name",
    albumName: "Album Title",
    genre: "Pop",
    releaseYear: 2024,
  },
]

export default function Demo() {
  return (
    <AudioSourceProvider source={DEMO_SOURCE}>
      <div>音频源已加载</div>
    </AudioSourceProvider>
  )
}
```

<h2 id="api">API</h2>

### AudioSourceProvider

音频源提供者主组件，负责将音频资源数据注入 LimePlay 播放器 Store。

| 属性 | 说明 | 类型 | 默认值 |
| --- | --- | --- | --- |
| source | 音频资源数据，支持字符串 URL、单个 Asset 或 Asset 数组 | `PlayerSource&lt;AudioPlayerAsset&gt;` | - |
| sourceKey | 资源唯一标识键，用于区分多个 AudioSourceProvider 实例 | `string` | - |
| autoLoad | 是否自动加载媒体源 | `boolean` | `true` |
| initialIndex | 初始播放项的索引 | `number` | - |
| loading | 自定义加载配置，支持 `getAssetId`、`recover`、`resolveSource` 等钩子 | `UseAssetOptions&lt;AudioPlayerAsset&gt;` | - |
| children | 子节点，在资源注入后渲染 | `ReactNode` | - |

### AudioPlayerAsset

单个音频资产的完整数据结构。

| 属性 | 说明 | 类型 |
| --- | --- | --- |
| id | 资产唯一标识，用于播放列表去重和恢复 | `string` |
| src | 直接播放 URL | `string` |
| title | 曲目标题 | `string` |
| artistName | 艺术家名称 | `string` |
| albumName | 专辑名称 | `string` |
| genre | 音乐流派 | `string` |
| releaseYear / year | 发行年份 | `number \| string` |
| poster | 封面海报图 URL | `string` |
| images | 图片集（包含 `poster` 和 `backdrop`） | `{ poster?: string; backdrop?: string }` |
| artwork | 艺术作品（包含 `url` 和 `templateUrl`） | `{ url?: string; templateUrl?: string }` |
| description | 描述文本 | `string` |
| duration | 时长（秒） | `number` |
| features | 特性标签数组 | `string[]` |
| group | 分组名称 | `string` |
| playbackUrls | 需要二次请求解析的播放 URL 对象 | `{ primary: string; secondary?: string }` |
| config | Shaka Player 配置 | `shaka.extern.PlayerConfiguration` |

### PlaybackUrls

用于需要异步解析的播放地址场景（如需要请求签名或 token 的服务端播放地址）。

| 属性 | 说明 | 类型 |
| --- | --- | --- |
| primary | 主播放 URL，组件会向该地址发起请求以获取实际播放链接 | `string` |
| secondary | 备用播放 URL | `string` |

当资产同时设置了 `src` 和 `playbackUrls` 时，优先使用 `src`。

### getAudioAssetMetadata

工具函数，用于从 `AudioPlayerAsset` 中提取显示元数据。

```ts
function getAudioAssetMetadata(
  asset: AudioPlayerAsset | null | undefined,
  fallbackTitle?: string
): AudioAssetDisplayMetadata
```

返回的 `AudioAssetDisplayMetadata` 包含 `title`、`subtitle` 和 `poster` 三个字段，其值按优先级从资产的多个字段中自动推算。

### 数据流说明

AudioSourceProvider 内部使用 `PlaybackSourceController` 处理数据：

1. `source` 被标准化为资产数组（字符串转 `[{ src }]`，单个对象转 `[{ ...obj }]`）。
2. 资产通过 `usePlaylist` hook 加载到播放列表 Store。
3. 首个资产通过 `usePlaybackStore` 触发播放器加载。
4. 加载错误和播放错误通过 `recover` 配置处理（默认：加载错误时 Skip 到下一首，网络错误时 Reload）。

<h2 id="faq">常见问题</h2>

### AudioSourceProvider 和 source 直接传 URL 有什么区别？

AudioSourceProvider 支持更丰富的数据结构（`AudioPlayerAsset`），可以携带专辑名、艺术家、封面等元数据。直接传 URL 仅能作为最简单用法，缺少元数据时播放器 UI 将无法展示详细信息。

### 如何实现多源切换？

通过 `sourceKey` 区分不同的 AudioSourceProvider 实例。在播放器切换音源时，更换 `sourceKey` 的值，组件将重新加载新的资源集合。

### 播放列表如何读取？

子组件通过 `usePlaylistStore` 读取播放队列：

```tsx
const queue = usePlaylistStore((state) => state.queue)
const currentItem = usePlaylistStore((state) => state.currentItem)
```

或通过 `usePlaylist` hook 获取带操作方法的播放列表接口：

```tsx
const { queue, currentItem, load, append } = usePlaylist()
```

### 如何自定义加载行为和错误处理？

通过 `loading` 属性传入自定义的 `UseAssetOptions`：

```tsx
<AudioSourceProvider
  source={sources}
  loading={{
    getAssetId: (asset) => asset.myCustomId,
    recover: {
      loadError: (asset, error, { hasNext }) => {
        return hasNext ? AssetRecoveryAction.Skip : AssetRecoveryAction.Stop
      },
    },
    resolveSource: async ({ asset, signal }) => {
      const response = await fetch(`/api/resolve/${asset.id}`, { signal })
      const { url } = await response.json()
      return { src: url }
    },
  }}
/>
```

### 为什么使用 playbackUrls 而非 src？

当播放地址需要服务端动态签名或存在过期机制时，使用 `playbackUrls.primary` 提供获取真实播放地址的 API 端点。组件会在需要时自动请求该端点获取实际的播放 URL。

<h2 id="llm-rules">LLM 使用规则</h2>

1. 从 `@/components/ui/audio-source` 导入 `AudioSourceProvider`。
2. 必须在 `MediaProvider` 上下文中使用，否则 Store 不可用。
3. `source` 支持三种形式：字符串 URL、单个 AudioPlayerAsset 对象、AudioPlayerAsset 数组。
4. 每个 AudioPlayerAsset 至少需要 `src` 或 `playbackUrls.primary` 之一。
5. 如需携带元数据（标题、艺术家、封面等），使用完整的 AudioPlayerAsset 对象。
6. 不要引入 Ant Design 相关依赖。
7. 子组件可通过 `usePlaylistStore` / `usePlaybackStore` 读取注入的资源数据。
8. 多源场景可通过 `sourceKey` 区分实例。
