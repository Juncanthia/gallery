# AutoComplete 自动完成

输入框自动完成功能。

本组件借鉴 Ant Design AutoComplete 的使用心智，但不是 AntD 兼容层。实现以本地源码为准，不引入 AntD 的 ConfigProvider、css-in-js、Design Token 运行时或历史兼容语法糖。

底层基于 `Command`（cmdk）+ `Popover` 直接实现（input-first：输入框本身就是弹层锚点），不再依赖 `Select`。这与 `Combobox`（`@/components/core/combobox`，按钮触发、只能从列表中选择）是两种不同交互模型：AutoComplete 允许自由文本，值不要求匹配任何选项。

## 何时使用 {#when-to-use}

适合在需要 自动完成 相关能力的界面中使用。优先通过组件自身 props 表达状态和语义，避免在业务侧堆叠重复 className。

常见场景：

- 需要复用统一设计语言、状态样式和可访问性约定。
- 需要和本项目 Tailwind 主题变量、暗色模式、Radix/Base UI primitive 或本地交互行为保持一致。
- 需要让 AI 或业务代码稳定地从 `@/components/core/auto-complete` 引用组件，而不是直接引用第三方底层库。

不推荐：

- 为了照搬 Ant Design API 而新增兼容别名。
- 在调用侧绕过组件 API，直接依赖内部 DOM 结构完成业务逻辑。
- 引入 `antd`、`@ant-design/icons`、`antd-style` 或 rc-component 作为使用该组件的前置条件。

## 代码演示 {#examples}

### 基础用法

```tsx
import { AutoComplete } from "@/components/core/auto-complete"

export default function Demo() {
  return (
    <AutoComplete>
      自动完成
    </AutoComplete>
  )
}
```

> 说明：这是面向 LLM 的最小导入示例。复杂组件应结合下方 API 摘要和源码类型补充必需 props，例如 `options`、`items`、`columns`、`dataSource` 等。

## API {#api}

### AutoComplete

| 属性 | 说明 | 类型 | 默认值 |
| --- | --- | --- | --- |
| options | 输入时建议的可选项列表，支持平铺和分组格式 | `AutoCompleteOptionItem[]` | - |
| placeholder | 未输入值时的占位文本 | `React.ReactNode` | - |
| size | 输入框尺寸 | `AutoCompleteSize` | `"middle"` |
| variant | 视觉变体 | `AutoCompleteVariant` | `"outlined"` |
| blocks/status | 校验状态 | `AutoCompleteStatus` | - |
| disabled | 是否禁用输入框 | `boolean` | - |
| allowClear | 值非空时是否显示清除按钮 | `boolean` | - |
| prefix | 输入框前渲染的内容 | `React.ReactNode` | - |
| suffixIcon | 自定义后缀图标，传入 `null` 隐藏 | `React.ReactNode \| null` | - |
| notFoundContent | 无匹配选项时展示的内容 | `React.ReactNode` | `"No results found."` |
| optionRender | 每个选项的自定义渲染 | `(option: AutoCompleteOption) => React.ReactNode` | - |
| popupRender | 包装弹出列表节点（如追加底部按钮） | `(originNode: React.ReactNode) => React.ReactNode` | - |
| placement | 弹出层相对触发器的位置 | `"bottomLeft" \| "bottomRight" \| "topLeft" \| "topRight"` | `"bottomLeft"` |
| filterOption | 过滤器函数或布尔值控制本地过滤，设为 `false` 用于异步搜索 | `boolean \| ((input: string, option: AutoCompleteOption) => boolean)` | `true` |
| onSearch | 输入值变化时的回调（用户键入），适合异步搜索场景 | `(value: string) => void` | - |
| onSelect | 从下拉列表选择选项时的回调 | `(value: string, option: AutoCompleteOption) => void` | - |
| onClear | 值被清除时的回调 | `() => void` | - |
| open | 受控的打开状态 | `boolean` | - |
| defaultOpen | 默认打开状态（非受控） | `boolean` | - |
| onOpenChange | 弹出层打开/关闭时的回调 | `(open: boolean) => void` | - |
| value | 受控值（自由文本输入，不限于选项） | `string` | - |
| defaultValue | 默认值（非受控） | `string` | - |
| onValueChange | 值变化时的回调，每次按键及选择选项时触发 | `(value: string \| undefined) => void` | - |
| onChange | onValueChange 的别名，兼容 AntD 命名 | `(value: string \| undefined) => void` | - |
| className | 根触发器元素的 CSS 类名 | `string` | - |
| contentClassName | 弹出内容区域的 CSS 类名 | `string` | - |
| listClassName | 选项列表的 CSS 类名 | `string` | - |
| name | 底层表单 input 的 name 属性 | `string` | - |
| id | input 元素的 id 属性 | `string` | - |
| required | 表单上下文中是否为必填 | `boolean` | - |
| aria-label | input 的无障碍标签 | `string` | - |

> 相比历史版本，已移除 `labelRender`（自由文本输入没有"选中项"概念，无法可靠地把 React 节点渲染进原生 `<input>`）和 `filterSort`（未被使用）。

### 派生类型

- `AutoCompleteOption` — 选项类型：`{ label, value, disabled?, description?, keywords?, group? }`
- `AutoCompleteOptionGroup` — 选项分组类型：`{ label, options }`
- `AutoCompleteOptionItem` — `AutoCompleteOption | AutoCompleteOptionGroup`
- `AutoCompletePlacement` — `"bottomLeft" | "bottomRight" | "topLeft" | "topRight"`
- `AutoCompleteSize` — 尺寸类型（复用 `@/components/core/select-variants` 的 `SelectSize`）
- `AutoCompleteStatus` — 状态类型（复用 `SelectStatus`）
- `AutoCompleteVariant` — 变体类型（复用 `SelectVariant`）

## Semantic DOM {#semantic-dom}

源码中出现的稳定语义槽位：

- `data-slot="auto-complete"` — 根触发器容器（`role="combobox"`，`aria-expanded`）。

额外状态属性：

- `aria-disabled` — 组件被禁用时。
- `aria-expanded` — 弹出列表是否展开。

生成测试、样式定位或自动化脚本时，优先使用这些语义属性；不要依赖易变的 Tailwind class 拼接顺序。

## 样式模型 {#styling-model}

- 样式主要由 Tailwind CSS className 与本地 `cn` 合并函数控制。
- 如果源码使用 CVA，则变体样式集中在组件文件或相邻 `*-variants.ts` 中。
- 如果源码包装 Radix/Base UI primitive，则交互状态、键盘行为和 ARIA 基础能力来自底层 primitive，本地组件负责视觉和 API 收敛。
- 主题颜色优先使用 `bg-background`、`text-foreground`、`border-border`、`text-muted-foreground`、`ring-ring` 等项目级语义变量。

## 依赖与实现说明 {#implementation-notes}

核心文件：

- `src/components/core/auto-complete.tsx`

直接依赖摘要：

- `@/components/core/command`（`Command`、`CommandList`、`CommandGroup`、`CommandItem`、`CommandEmpty`，以及直接从 `cmdk` 导入的 `Command.Input` 用于自定义触发器样式）
- `@/components/core/popover`（`Popover`、`PopoverAnchor`、`PopoverContent`）
- `@/components/core/select-variants`（`selectTriggerVariants`、`selectInputVariants` 及其类型，仅复用视觉样式，不依赖 `Select` 组件本身）

实现注意：

1. 组件 API 以本地 TypeScript 类型为准。
2. 样式应优先通过组件 props、variant 或本地 className 扩展点表达。
3. 如果需要新增能力，应先修改该组件源码和相邻 variant 文件，再更新本 llm 文档。
4. 不要把第三方文档中的全量 API 视为本地已支持能力。

## 可访问性 {#accessibility}

- 保留原生元素或底层 primitive 的键盘语义。
- 交互型组件应提供可访问名称，例如 `aria-label`、`title` 或可见文本。
- 禁用、加载、展开、选中等状态应优先使用组件已有 props，而不是只改视觉 class。
- 纯图标触发器必须提供明确的可访问名称。

## 与 Ant Design 的取舍 {#antd-differences}

| 方向 | 本地决策 |
| --- | --- |
| API 来源 | 以 `src/components/core/auto-complete.tsx` 的 TypeScript 类型为准 |
| 视觉系统 | 使用 Tailwind CSS 和项目主题变量 |
| 全局配置 | 不依赖 AntD ConfigProvider |
| 样式运行时 | 不引入 AntD css-in-js 或 token 运行时 |
| 兼容目标 | 借鉴 AntD 使用心智，但不承诺 API 兼容 |

## LLM 使用规则 {#llm-rules}

当 AI 生成本项目 AutoComplete 代码时，应遵守：

1. 从 `@/components/core/auto-complete` 导入，不要从 `antd` 导入。
2. 只使用本文件 API 摘要和源码中存在的 props。
3. 不要生成 AntD 专属的 `classNames`、`styles`、`ConfigProvider`、Design Token 配置，除非本地源码明确支持。
4. 需要自定义样式时，优先使用组件暴露的 props；最后才使用 `className`。
5. 交互组件必须保留键盘和 ARIA 语义，纯图标操作必须补 `aria-label`。
6. 如果不确定某个 AntD API 是否存在，先检查 `src/components/core/auto-complete.tsx`，不要凭 AntD 记忆生成。
